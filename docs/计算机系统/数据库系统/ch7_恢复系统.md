# 恢复系统

## 故障恢复

计算机系统像任何其他系统一样会发生故障，一旦任何故障发生，就有可能丢失信息，因此数据库必须采用预先措施，即恢复(recovery)机制。

### 故障分类

* 事务错误 
  * 逻辑错误：事务由于某些内部情况而无法继续执行，比如溢出、输入错误、未找到数据等。
  * 系统错误：系统进入一种不良状态，比如死锁。
* 系统崩溃 ：硬件故障，或者是数据库软件或者操作系统漏洞，但是假设非易失性存储里面的数据不会丢失。
* 磁盘故障 ：在数据传输的过程中由于磁头损坏或故障造成磁盘块上的内容丢失。

### 恢复算法

恢复算法有两部分：

1. 正常执行的事务处理需要能够留存足够多的信息，即日志。
2. 在事务失败之后恢复数据，保证原子性、一致性和持久性。

### Data Access

数据库系统常驻留于非易失性存储器，在任何时候都只有数据库的部分内容在内存中。在磁盘和系统主存之间传输数据是通过`input(B)`和`output(B)`。逻辑上每个事务都有自己独立的工作区，在这个工作区中会复制它需要的数据，在系统主存和事务的工作区域之间传输数据是通过`read(X)`和`write(X)`。

## 基于日志的恢复

### 修改策略

日志是数据库系统的“流水账”，日志被保留在stable storage中，要记录事务的开始、提交和终止。概念上有两种修改数据库的策略，不过要注意所谓的修改数据库指的是修改主存中的块缓存或者磁盘上的块，修改事务自己复制的数据不算修改数据库：

1. 延迟修改：在事务提交之前不会对数据库进行任何的修改，因而更改日志只需要记录新值，恢复操作也只有redo。
2. 立即修改：在事务提交前后都可以修改数据库，因而更改日志需要记录旧值和新值，恢复操作有undo和redo。

本章讲解的所有恢复算法都是以立即修改为例，不过对于延迟修改也适用。

### 日志格式

更新日志的格式为$<T_i, X_j, V_1, V_2>$，四个数依次表示事务编号，数据项编号（通常会有页ID和偏移量），旧值，新值。此外还有三种用于记录事务状态的日志，分别是`begin`，`commit`和`abort`。

### 恢复流程

redo是从上到下扫描一遍日志并把所有操作都依次置为新值，undo是从下往上把未提交的事务更新过的数据项都重置为旧值，这个过程需要写补偿日志(redo-only，不需要包含新数据项的旧值，redo这些日志相当于是在undo)，撤销完一个事务的操作后需要写`abort`，这个也算事务结束了。

### 检查点

搜索整个日志的开销是很大的，因为日志很离谱的多，而且大多数事务其实都已经提交了，为了降低恢复时的开销，引入检查点。检查点就是可以先把已经完成提交的事务全部都写入稳定存储，再把脏页都刷回磁盘，然后可以把这部分事务的日志从日志缓存移除，记录写检查点时还活跃的事务，下次崩溃恢复只需要从这些事务开始重做。写检查点的时候系统会停机，不允许事务执行任何更新操作，为了减少停机的影响，后面有模糊检查点。

### 恢复算法

#### 事务回滚

这是正常操作时事务回滚，回滚$T_i $：从后往前扫描日志，undo所有$T_i $相关的日志并写补偿日志，一旦发现$T_i$ start则停止扫描，并写abort

#### 系统崩溃后恢复

1. 重做阶段：从最后一个检查点正向扫描来重演所有的事务更新，重演包含之前系统崩溃之前已经回滚的事务和崩溃时还没提交的事务的日志。这个阶段还需要确认有哪些事务需要回滚，在开始扫描前现将`undo_list`初始化为检查点中的列表`L`，一旦发现$T_i $ start就将$T_i $添加到`undo_list`中，一旦发现$T_i $ commit/abort 就把$T_i $从`undo_list`中删除。 
2. 撤销阶段：系统回滚`undo_list`中的所有事物，从尾部开始反向扫描，一旦发现`undo_list`中相关事务的日志就执行撤销操作，并写补偿日志，当系统发现$T_i $ start时就写$T_i $ abort，持续这个过程直至`undo_list`为空。

## 缓冲区管理

### 日志记录缓冲

通常，写稳定存储是以块为单位，因此日志需要在主存中有自己的缓冲区，因为日志可能有较长时间都驻留在主存中，由于在系统崩溃时这种日志会丢失，所以必须对恢复技术增加一些额外的要求以保证事务的原子性，有如下的四条规则：

1. 日志必须按在缓冲中的顺序输出到稳定存储。
2. T_i $进入提交状态当且仅当$<$T_i$  commit> 被输出到稳定存储。
3. 在$<T_icommit> $能够被输出之前，所有与$T_i $相关的日志必须被输出到稳定存储。
4. 日志应先于数据写到磁盘（write-ahead logging rule）

### 数据库缓冲

数据库维护了一个主存中的 缓冲，按照之前所学，当需要一个新的块的时候，如果缓冲区满了，按照LRU的策略，应该移除最近最少使用的块，如果这个块已经被更新过了，则应该写回磁盘。根据日志先行的原则，当一个还未提交的块被写到磁盘，应该首先写回所有和它相关的日志。在写回$B_i $的过程中不应该对它有任何更新，这个可以被以下的三点保证：

1. 在写数据之前，事务应该请求$B_i $的互斥锁
2. 这个锁应该在写完了再释放

数据库的缓存既可以实现为一段被预留出来的真实空间，也可以是虚拟内存(virtual memory)，实现为真实的空间有以下缺陷：

1. 内存已经被数据库缓冲和其他应用划分了，限制了灵活性，因为内存的分配实际上是操作系统在负责
2. 需求可能会改变，虽然操作系统知道如何最好地划分空间，但是它不能修改内存的划分

当操作系统为了给另一个页腾出空间，需要输出一个被改变的页时，这个页会被写进交换空间，但是当数据库决定写回缓冲页时，缓存页可能在交换空间里面，需要从硬盘上的交换空间读出来，再写回数据库的磁盘，造成了额外的I/O开销，这一现象被称作 dual paging problem，所以高效的DBMS系统会自己做内存管理。

### 模糊检查点

为了避免在写检查点时长时间的打断正常进程，应该要允许写检查点时更新，实际上是节省了刷回脏页的时间:

1. 暂时停止所有事务的更新，还是要先短暂停止
2. 写检查点`checkpoint L`和force log to stable storage
3. 标记被修改过的buffer block为集合M
4. 现在允许事务继续执行它们的行为
5. 把M中的所有buffer都写到硬盘
6. 保存一个指向最近的检查点的指针，如果在写检查点的过程中失败则这个指针还是指向上一个检查点

### 非易失存储器数据丢失故障

非易失性存储也有可能发生故障，需要定期转储(dump)数据库中的所有信息到稳定存储，dump比checkpoint更耗时，所以也可以fuzzy dump，实际上就是备份整个数据库。

## 更早的锁释放

### 简介

高并发的前提下锁尽早释放有助于提高并发度，比如在B+树中。

### 逻辑撤销

像B+树插入这种，在执行完之后你已经释放对应的锁了，要撤销应该执行一次对应的删除，这个是与写回旧值的物理重做相比。在这种情况下undo操作需要包含具体执行的操作。

### 操作日志

对每一步的操作记录日志：

1. 当一个操作开始了，写$<T_i,O_j,operation-begin> $
2. 操作执行过程中写物理重做和撤销日志
3. 操作结束，写$<T_i,O_j,operation-end, U> $，其中$U $里面包含信息用于逻辑撤销

如果在操作结束之前发生了崩溃/回滚，那么就物理撤销，否则使用逻辑撤销，redo始终都是物理的。

## ARIES

### 简介

ARIES是真正的实现，**考试必考**，前面仅仅是一些原理，它有以下特点：

1. 使用 LSN 来识别记录
2. 物理逻辑重做
3. DPT用于避免没必要的redo
4. 模糊检查点只记录脏页，而不要求一下写回所有脏页

虽然上一小节说的不存在物理逻辑的重做，但是这里确实可以，因为ARIES受影响的页被物理标记了，页内的操作可以是逻辑的：

1. 用于减少日志的开销：比如一页中删掉一条记录，然后用其他的记录补全，如果要记录物理日志就需要写很多很多页的，但是用逻辑日志就只需要记录删除操作。
2. 需要页被自动输出到磁盘：确保数据库里面的数据是最新的

### 相关数据结构

它使用到了一些数据结构：

1. LSN(log sequence number)：给record一个标号，要求是顺序增长的。
2. Page LSN：最近一条写该page的事务对应的日志的LSN。
3. Recent LSN：对于一个page，自上一次刷盘以来，第一次操作对应的日志的LSN。
4. Previous LSN：同一个事务中后一条日志会把前一条日志的LSN记录下来，形成一个反向链表，便于Undo时回溯事务。
5. CLR(compensation log record)：补偿日志，redo-only的日志，由于ARIES采用逻辑撤销，撤销是不幂等的，不可以重复执行。
6. Undo Next LSN：用于补偿日志，记录下一条（更早的）应该undo的记录
7. Dirty Page Table：缓冲中被修改过的页的列表，每一页都包含Page LSN、Rec LSN
8. Checkpoint Log Record：包含DPT和活跃中的事务，磁盘上的固定位置会标记上一个完成了的检查点

### 算法

ARIES 恢复算法包含三个阶段：

1. 分析阶段：
   1. 有哪些事务要 undo
   2. 有哪些页是脏的
   3. 从哪条记录开始redo，确定Redo LSN
2. 重做阶段：重做从 Redo LSN 开始所有记录
3. 撤销阶段：撤销所有不完整的事务
