# 并发控制

## 两阶段加锁

锁(lock)是用于控制并发访问的机制，我们有两种基础的锁：

- exclusive(X) 排他锁，也就是写锁，不允许任何两个事物进行同时操作
- shared(S) 共享锁，也就是读锁，虽然不能同时写，但是可以同时读

和上一章一样，读锁是可以共存的，但是写锁和写锁，写锁和读锁都是不能共存的，如果一个进程想要获得的数据已经有不兼容的锁了，那么它就必须等待所有不兼容的锁被释放。

### 两阶段加锁协议

两阶段加锁协议可以保证冲突可串调度：

- 阶段一：Growing Phase 可以获得锁，但是不能释放锁
- 阶段二：Shrinking Phase 可以释放锁，但是不能获得锁

两阶段加锁协议对应的串行调度保持锁点（一个事务获得最后一个锁）一致，可以使用反证法或者归纳法。

两阶段加锁协议还有两种扩展，大部分的数据库系统实现了强两阶段加锁协议：

- strict two-phase locking 严格两阶段加锁：一个事务必须持有它的所有排它锁直到它提交/释放，好处是保证可恢复性和避免级联回滚。
- rigorous two-phase locking 强两阶段加锁：一个事务必须持有它的所有锁直到它提交/释放。

数据库一般默认实现了强两阶段加锁协议，但是只称作两阶段加锁协议，两阶段加锁协议保证可串行，但是可串行不一定需要两阶段加锁。

### 锁转换

两阶段加锁协议带有锁转换，这个协议也是保证可串行化：

- 第一个阶段可以获得锁，或者把读锁升级到写锁
  - 如果要获得共享锁，首先要检查事物有没有锁，如果有则可以直接返回；没有则先检查这个数据项有没有互斥锁，如果存在互斥锁则需要进入等待，一直等到可以加共享锁。
  - 如果要获得互斥锁，首先还是检查事物有没有互斥锁，没有的情况下，如果有共享锁则升级，没有则赋予互斥锁，前提是这个数据项没有其他任何锁。

- 第二个阶段可以释放锁，或者把写锁降级到读锁

### 锁管理

锁可以用单独的 lock manager 管理，事务可以向它发送加锁和解锁的请求，如果一个事务向锁管理器发起了锁的申请，就会一直等到锁管理器返回授权锁的信息或者回滚的信息，锁管理器会记录分发出去的锁和正在等待的请求。

通常是用一个内存中的哈希表实现，对数据项的编号或者地址进行哈希，每个数据项又会维护一个请求队列，队列里面尾巴上是已获得锁的事务，头是还在等待的事务。

## 死锁控制

如果两个事物分别持有对方需要的不兼容的锁，那么就发生了死锁，两阶段加锁协议并不能保证不发生死锁。

### 避免死锁

死锁防控机制(deadlock prevention protocol)保证系统绝对不会进入死锁，一个事务在执行前获得它需要的所有锁，但是这样会降低串行度，还可以对所有的事务施加偏序（基于图的加锁协议），这种情况下一个事务必须按照偏序上锁。

解决死锁可以指定一个等待时间的上限，如果等待超时则回滚事务，这样很容易实现，但是可能会饿死(starvation)，就是同一个事物一直回滚没法执行，并且时间区间选择多少也比较麻烦。

死锁检测可以用一个等待图(wait-for graph)，这个图是在事物执行过程中不断变化的，做过实验知道并不需要维护一个等待图，而是应该在唤起时构建等待图，顶点是事务，如果事务$T_i$正在等待被$T_j$持有的锁(或其他资源)释放，则画一条从$T_i$到$T_j$的边，一个系统会发生死锁当且仅当等待图有环。当死锁被检测到了，一些事务需要回滚，回滚可以是完全回滚，但是更好的肯定是部分回滚（只回滚有必要的部分，后面ARIES可以支持），为了避免饿死，还需要记录每个事物的回滚的次数，不能每次都回滚同一个事务。

### 基于图的协议

基于图的协议是两阶段加锁协议的替代方案，给不同的数据施加一个偏序（partial ordering），如果要对一个数据项上锁，则需要对它之前的所有数据项都上锁。进一步改进是基于树的加锁协议，树本身就是特殊的图，只允许互斥锁，如果要对一个数据项加锁，则需要对它的所有父节点都加锁。

### 多粒度加锁

如果要对一个数据项加锁，比如我要大范围地修改一个表，没有必要对表中的每条 tuple 都加锁，而是可以直接对这张表加锁，多粒度也可以表示成一颗树，不要与基于树的加锁协议混淆。但是如果要对一个节点加锁，则需要检查它的所有子节点的状态，这是很麻烦的事情，所以引入了意向锁（就是你有对它的子节点加锁的意向）

在一个节点被显式锁住之前，意向锁会被施加到它的所有的父节点上，意向锁允许更高级别的节点并以S或者X锁的形式被锁住，而不需要去检查它的子孙节点。这里有三种意向锁：

1. IS 表明其后代存在 S 锁
2. IX 表明其后代存在X锁
3. SIX = S + IX

综上，最终版本的多粒度锁是加锁自顶向下，解锁自下而上，且遵守2PL协议，必须满足锁的兼容性矩阵。

### 其他操作加锁

如果使用两阶段加锁协议，删除操作必须有对应元组的互斥锁，插入也必须有互斥锁。插入和更新可能会导致幽灵现象，如果只用元组级别的锁，可能会导致非冲突可串。一个解决方案是添加一个数据结构包含这个表的所有信息，扫描表的需要对数据项持有读锁，而插入或者删除需要写锁，这样做的并发度很低。更高并发度的方案有索引加锁协议，但是要求也很多：

1. 每张表都必须至少有一个索引
2. 访问表必须至少通过一个索引
3. 一个查找的事务必须对它访问到的所有叶节点加读锁
4. 插入、删除、更新元组都必然修改索引，操作前必须获取所有受影响叶节点的写锁
5. 必须遵守两阶段加锁协议
