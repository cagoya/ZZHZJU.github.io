# 关系型数据库的设计
## ER 模型
E是Entity，表示一系列的属性，Relationship表示不同实体之间的联系，实体集是矩形，联系是菱形，联系可以有自己的属性。
### ER 图
#### 度
一般是二元关系，有时也会是三元

#### 映射基数
表示是几对几的关系，单箭头是单，没有箭头是多；也可以写`0..*`和`1..1`这种

#### 参与度
* 全参与：每条tuple都有对应关系
* 部分参与：可以有不参与关系的tuple

#### 弱实体集
没有主键，弱实体集依赖于标识性实体集，必然有全参与、一对多的关系（从标识性实体集到弱实体集）。虽然没有主键，但是有分辨符(discriminator)。弱实体集和标识关系都是双层的边框。

#### 转化成表
1. 强实体集直接转换
2. 弱实体集要包含标识集的主键
3. 多对一，一对多在多的那一侧加主键，多对多两边都必须有主键

### 拓展的 ER 特征
#### 层和泛化
* Specialization：自顶向下的设计，低层级的会继承高层级的关系，类似于OOP的继承
* Generalization：自底向上的设计，把低层级的共享特征的实体合并到上层

层和泛化是反转的关系，它们在ER图中有相同的表示，这里有两个主要的问题：
1. 什么样的实体能够到更低的实体集：
   1. 条件定义的
   2. 用户定义的
2. 一个实体是否能够属于多个实体集
   1. 不相交 disjoint 应该合并箭头
   2. 可重叠 overlapping 应该画两条箭头分别指上去
3. 完全性约束，高层级的实体是否必须属于低层级的实体集
   1. total 完全泛化 两条线
   2. partial 部分泛化

#### 聚集
类似于 OOP 的聚合，可以把一些实体集当作一个整体（即复合实体集）

### 关于 ER 的思考
* 如果一个对象只对其名字及单值感兴趣可作为属性，若除了名字之外，还有其他属性需要描述，则应该定义为实体集
* 是作为实体集还是关系集取决于映射基数，比如顾客和银行之间，如果顾客可以有多笔贷款，那么就不能作为联系集
* 是作为实体的属性还是关系集的属性，要从对象语义独立性和减少数据冗余考虑
* 是用二元关系还是多元关系

## 范式
### 第一范式
所有的域都是原子的，所有的关系型数据库中的表都必须满足第一范式，不满足原子性会导致复杂重复的存储并且不便于查询：
* 重复的存储，浪费了空间
* 更新的开销很大
* 插入/删除可能需要引入null，但是null难以管理

原子性实际上是关乎这个域的属性怎么用，比如字符串虽然通常是不可分的，但是极端的例子可以去子字符串提取信息，这是不好的设计。

如何处理非原子的属性：
* 对于复合属性，使用一系列的属性
* 对于多值属性：
  * 使用多个域
  * 使用独立的表
  * 使用单一的域

### 分解
将一张表拆成多张表，主要的要求是无损连接(lossless join)和依赖保持(dependency preservation)，无损连接是指分解后的表可以拼接出分解前的表，依赖保持指的是每个子关系的函数依赖闭包加起来的闭包等价于分解前的闭包。

### 函数依赖
#### 定义
函数依赖借鉴了数学上的函数，指的是如果$t_1[\alpha]=t_2[\alpha] $，则$t_1[\beta]=t_2[\beta] $，函数依赖是一致性约束的一种，是对键的泛化：
* 主键：$K $是主键等价于$K\rightarrow R$
* 候选键：$K $是候选键等价于$K\rightarrow R$，并且不存在$K $的真子集能够决定$R $

对于一个给定具体数据的表，判断它是否满足函数依赖是容易的，但是对于一张抽象的表，判断是否满足函数关系几乎是不可能的，函数依赖通常是在定义表时给出的。

#### 公理
对于任意关系都满足的函数依赖被称作是平凡的(trivial)，即属性集合可以决定自己的子集

三条基本公理：
* 自反律
* 增补律
* 传递律

三条扩展的公理：
* 合并律
* 分解律
* 伪传递律

#### 闭包
函数依赖的集合记为$F $，它的闭包记作$F^+ $，具体的算法为：
```
Result = F
重复：
   对于Result中的所有关系使用自反律和增补律，能够用传递律的用传递律，把新增的函数依赖添加到 Result 中
直到 Result 中的内容不再增加
```

可以用$F^+ $去判断主键，但是这个算法的开销可能很大，理论上最多有$2^n\times 2^n $种函数依赖，所以引入了属性的闭包，记作$\alpha^+ $，具体的算法为
```
Result = \alpha
重复：
   遍历整个集合，如果存在 \beta \rightarrow \gamma，并且\beta在 Result中，则将 \gamma 添加到Result 中
直到 Result 中的内容不再增加
```

#### 正则覆盖
正则覆盖指的是等价于$F $的最小函数依赖集合，记作$F_c $，构造正则覆盖的核心是去除重复，有三种重复的可能
1. 根据传递律$A\rightarrow B,B\rightarrow C $，其实是包含了$A\rightarrow C $的
2. 左边有多余
3. 右边有多余

寻找正则覆盖的具体算法是
```
重复：
   使用合并律（这一步其实能够解决第一种多余），然后寻找是否有函数依赖的左边或者右边有多余。
直到没有多余
```
判断一个元素集A是否多余：
* 左边：如果$((\alpha-A)\rightarrow B)\cup (F-(\alpha\rightarrow \beta)) $包含了$\alpha\rightarrow \beta $则$A $多余
* 右边：如果在只有$(\alpha\rightarrow (\beta-A))\cup (F-(\alpha\rightarrow \beta)) $的情况下$\alpha^+ $仍然包含$A $，则$A $多余

### BCNF
BCNF 范式要求$\alpha\rightarrow \beta $要么是平凡的，要么$\alpha $是超键，判断是否是超键可以使用$\alpha^+ $。注意在F上判断R是否满足BCNF是可行的，但是**在F上判断分解后的R是否满足BCNF范式是不可行的，必须使用$F^+$** BCNF的具体分解算法
```
Result = {R}
done = false
compute F^+
如果有不满足BCNF的，则将R_i分解程两个子关系：\alpha\rightarrow\beta是违反BCNF的，并且\alpha\cap\beta=\empty，则result = (result - R_i)\cup(\alpha,\beta)\cup(R_i-\beta)
```
BCNF的分解保证无损连接，但是BCNF不一定能依赖保持。

### 第三范式
允许一定重复，比BCNF稍弱，但是保证了依赖保持，比BCNF多一个允许条件：$\beta-\alpha $包含在候选键中，测试是否满足第三范式反而变难了，因为需要找出所有候选键，整个算法是NP hard的，但是计算第三范式的分解是可以在多项式时间内完成的，具体的分解算法为：
```
计算F_c
i = 0
对于F_c中的每一个函数依赖：
   如果没有任何的表包含\alpha和\beta，则
      i = i + 1;
      R_i = (\alpha, \beta) （保证依赖保持）
如果R_j(1<= j <=i)中没有任何一个包含R的主键：
   i = i + 1
   R_i = R的任何主键（保证无损连接）
```

### 多值依赖
多值依赖描述的是一种特殊的依赖关系，它表示一个属性的集合对另一个属性的集合是独立的，记作$\rightarrow\rightarrow $。函数依赖指的是一个属性确定另一个属性，多值依赖指的是一个属性决定另一个属性的取值集合。多值依赖的数学公式比较复杂：
$$
对于所有的t_1,t_2，存在t_3,t_4\\
t_1[\alpha]=t_2[\alpha]=t_3[\alpha]=t_4[\alpha]\\
t_3[\beta] = t_1[\beta]\\
t_4[\beta] = t_2[\beta]\\
t_3[R-\alpha-\beta] = t_2[R-\alpha-\beta]\\
t_4[R-\alpha-\beta] = t_1[R-\alpha-\beta]
$$

函数依赖其实是特殊的多值依赖，如果$\beta\subseteq \alpha $或者$\alpha\cup \beta=R $，则$\alpha\rightarrow\rightarrow \beta $是平凡的。

#### 第四范式
所有多值依赖$\alpha\rightarrow \beta $要么是平凡的，要么$\alpha $是超键，4NF一定是满足BCNF的。第四范式的分解算法和BCNF很像，就是改一下条件，既然BCNF都没法保证依赖保持，第四范式就更没法保证依赖保持了。
